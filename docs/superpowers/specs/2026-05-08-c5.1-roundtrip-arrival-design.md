# C5.1 — Round-trip Arrival Semantics

**Date:** 2026-05-08
**Owner:** @razinkele
**Status:** 🟡 v3 DRAFT — pass-2 review-loop closed (3 reviewers verified all 24 pass-1 findings closed; surfaced 13 NEW issues: 0 CRITICAL/BLOCKING, 3 HIGH, 5 MEDIUM, 5 LOW). v3 addresses all 13. Awaiting pass-3 verification (or user approval to proceed to writing-plans).

C5.1 tightens C5's arrival semantics from "in upper-quartile of natal
reach" (which counts both never-left agents and returned agents) to
"returned to upper-quartile of natal reach **after a verified
sea/coastal visit**" (which counts only the round-trip homing event).
Adds one boolean column to `AgentPool` (`been_to_sea`) and one new
event (`BeenToSeaEvent`); guards `ArrivalEvent`'s arrival mask with
the new column.

---

## §1 Why

C5 (v1.7.9) wired up `pool.arrived` after years of dead state. v1.7.10's
Playwright verify on laguna confirms the engine works — but exposes that
the metric is degenerate at this simulation horizon:

> Default scenario, rng_seed=42, n_agents=50, n_steps=480h:
> **46/50 alive · 46 arrived**, exit_branch_tagged=4.

Every alive agent reads as "arrived". Reason: spawning grounds are
intrinsically upstream-located, so agents start in cells that already
sit in the upper quartile of their natal reach. The 75th-percentile
threshold is a near-tautology with "alive in natal reach". The signal
the publication-grade four-tier hatchery-vs-wild architecture is
designed to produce is **homing return rate** (fraction of released
smolts that complete the migration loop and home back). C5 cannot
distinguish "started here, never left" from "left, came back".

C5.1 makes the metric biologically meaningful by requiring evidence of
a sea/coastal visit before an agent qualifies as arrived. The 480h
default is **likely too short** for substantial round-trip counts:
straight-line distance from inner Curonian Lagoon to BalticCoast via
Klaipėda Strait is ~50–100 km depending on natal reach, while
*effective* post-smolt net displacement after diel and passive-drift
components is typically **0.1–0.5 BL/s** (Thorpe & Ross 1981 measured
0.04–0.13 BL/s net downstream displacement in Loch Voil; Hedger et
al. 2008 documents predominantly nocturnal post-smolt swimming). The
1–2 BL/s figure that yields "30–60 km/day" appears in the literature
only as an upper-bound *active-swimming* rate (Lacroix & McCurdy 1996
tidal-aided; Solstorm et al. 2015 cage-preferred), not as a
sustained migration speed. Jutila et al. 2009 documents Simojoki
post-smolts reaching the southern Gulf of Bothnia "in late summer"
after spring sea-entry — weeks-to-months scale, not days.

After C5.1 the counter at the 480h horizon will read **likely 0** for
the default scenario (a few outlier agents that exit the lagoon early
might register), and that is the *correct* signal at this horizon.
Long-horizon (multi-year) simulations will produce the full
homing-return distribution. **Do not interpret arrived=0 at 480h as a
C5.1 bug; do interpret arrived≈n_alive as a C5.1 bug** (it would mean
the round-trip filter isn't firing).

## §2 Design

### Architecture

Round-trip semantics implemented as a **two-state guarded sticky flag**:

```
been_to_sea[i] := True iff agent i has, at some t' ≤ t, occupied a cell
                       whose reach_id ∈ at_sea_rid_set
                       (where at_sea_rid_set = {BalticCoast, OpenBaltic}).
arrived[i]     := True iff (been_to_sea[i]
                            AND in upper-quartile-of-natal-reach[i]).
```

Both flags are sticky-once-set; neither is ever cleared by any event.

### Why a SET, not just OpenBaltic

The Baltic salmon literature places the demographically dominant
post-smolt feeding phase in **coastal Baltic waters**, not offshore
(Friedland et al. 2016 — predation interactions concentrated "in
coastal waters and/or the upper water column typically occupied by
salmon post-smolts"; Jacobson et al. 2019 — multi-decadal post-smolt
distribution analysis). Strict OpenBaltic-only would treat a fish
that spent its entire post-smolt season in BalticCoast as "never been
to sea", which is biologically wrong and would give the metric
near-zero counts at any realistic simulation horizon. We err on the
inclusive side: BalticCoast cells count as "at sea" (see §7 on the
strictness defence).

### New state: `AgentPool.been_to_sea`

- `np.ndarray[bool]` of length `n_agents`, defaults all-`False` at init.
- Added to `AgentPool.ARRAY_FIELDS` so it round-trips through
  `compact()`, `add_agents()`, and the OOP `AgentView`.
- `add_agents()` initializes new agents with `been_to_sea = False`
  (mirrors the existing `arrived` field initialization at
  `population.py:275`).
- The `ARRAY_FIELDS` init guard at `agents.py:88-96` will assert this
  field is present in `__init__`.
- Mirrors `pool.arrived`'s shape and semantics exactly.

### New event: `BeenToSeaEvent`

Registered via `@register_event("been_to_sea")` in `events_builtin.py`.
Vectorised; reads agent's current cell `reach_id`, checks membership
in `_at_sea_rid_set`, sets `been_to_sea[i] = True` where the
cell-reach-id is in the at-sea set and `alive[i]` is True. Never clears.

```python
# Sketch (full code in plan):
class BeenToSeaEvent(Event):
    name: str = "been_to_sea"

    def execute(self, population, landscape, t, mask):
        sim = landscape["sim"]
        if not sim._is_baltic:
            return  # non-Baltic landscape; no-op
        # Direct .pool — fail-loud, matches existing ArrivalEvent
        # convention (events_builtin.py:569).
        pool = population.pool
        # Cast to int32 because mesh.reach_id is np.int8 and would wrap
        # at index 128. ArrivalEvent applies the same defensive cast
        # at events_builtin.py:591 for the same reason.
        cell_rid = sim.mesh.reach_id[pool.tri_idx].astype(np.int32)
        # Vectorised membership test in the at-sea reach set.
        # _at_sea_rid_set is a frozenset[int]; convert to ndarray for
        # np.isin to be fast on small sets (~2 entries).
        at_sea_rids = np.fromiter(sim._at_sea_rid_set, dtype=np.int32)
        at_sea_now = np.isin(cell_rid, at_sea_rids) & pool.alive
        # Sticky: only set True, never clear. Dead-and-already-True
        # agents stay True post-mortem (harmless; ArrivalEvent's
        # alive filter excludes them from arrival counting).
        pool.been_to_sea[at_sea_now] = True
```

### Guarded `ArrivalEvent`

`ArrivalEvent.execute` gains the `pool.been_to_sea` term in its
arrival mask. Existing C5 mask (verbatim from `events_builtin.py`):

```python
arrived_now = (
    in_upper_quartile_mask
    & pool.alive
    & ~pool.arrived
)
```

becomes:

```python
arrived_now = (
    in_upper_quartile_mask
    & pool.alive
    & pool.been_to_sea            # ← C5.1: round-trip guard
    & ~pool.arrived
)
```

Convention: keeps `pool = population.pool` direct (no `getattr` shim;
matches the existing fail-loud convention at events_builtin.py:569).
`BeenToSeaEvent` uses the same convention.

### Sim-init resolution: `_is_baltic` discriminator + `_at_sea_rid_set`

Two pieces of state cached at `Simulation.__init__`:

```python
# Positive Baltic-detection independent of any single reach lookup.
# Mirrors the discriminator pattern at delta_routing.py:178-187.
# Note: attribute is `self._species_config` (private; verified at
# simulation.py:302) — NOT `self.species_config`.
# Note: hatchery is included as an OR with wild because configs may
# legitimately have wild=BioParams (non-Baltic native pop) +
# hatchery=BalticBioParams (Baltic stocking experiment).
self._is_baltic = (
    (isinstance(self._species_config.wild, BalticBioParams)
     or isinstance(getattr(self._species_config, "hatchery", None),
                   BalticBioParams))
    and any(
        rn in self.mesh.reach_names
        for rn in delta_routing.BRANCH_FRACTIONS.keys()
    )
)

# At-sea reach IDs. ONLY constructed when Baltic-detected — no
# fallthrough default that could mask a typo. If Baltic-detected
# but neither reach name resolves, raise. If Baltic-detected and
# only ONE resolves (partial-miss — e.g., one of the two names was
# typo'd in the NC), log a WARNING per missing name AND continue
# with the partial set; do not raise (a partial set is still a
# usable round-trip filter, just narrower).
if self._is_baltic:
    rids = set()
    missing_names = []
    for name in ("BalticCoast", "OpenBaltic"):
        try:
            rids.add(int(self.mesh.reach_names.index(name)))
        except ValueError:
            missing_names.append(name)
    if missing_names and rids:
        # Partial miss — surface in logs, do not raise.
        logger.warning(
            f"c5.1-at-sea-reach-partial-miss: Baltic-detected "
            f"landscape resolved {sorted(rids)} but reach name(s) "
            f"{missing_names} were not found in mesh.reach_names. "
            f"Round-trip arrival proceeds on the partial set; "
            f"verify NC build if this is unexpected."
        )
    if not rids:
        # Total miss — fail loud at sim init.
        raise RuntimeError(
            f"{ERR_C5_1_AT_SEA_REACHES_MISSING}: Baltic-detected "
            f"landscape but neither 'BalticCoast' nor 'OpenBaltic' "
            f"is in mesh.reach_names={self.mesh.reach_names}. "
            f"Round-trip arrival cannot be computed. Likely cause: "
            f"the loaded NC was built before C5.1 added at-sea "
            f"reach assignment, OR a stale cached NC is shadowing "
            f"a freshly-rebuilt copy. Rebuild the H3 multires "
            f"landscape with the current build script and verify "
            f"the loaded NC path."
        )
    self._at_sea_rid_set = frozenset(rids)
else:
    self._at_sea_rid_set = frozenset()
```

The discriminator `_is_baltic` is **independent of the at-sea reach
lookup**, closing the single-point-of-failure pattern that produced
the v1.7.10 `getattr(pool, "n_arrived", 0)` bug. A *complete* miss
of both reach names raises a `RuntimeError` with explicit
reach-list output and likely-cause guidance. A *partial* miss of
one name logs a warning and proceeds (the round-trip filter is just
narrower; not a fatal misconfiguration). The `wild OR hatchery`
disjunction in the discriminator handles future configs where the
Baltic species_config arrives via the hatchery side only.

Wired in `simulation.py` after `assert_branch_topology(self.mesh)`
(simulation.py:306) and before `_compute_arrival_thresholds()`
(simulation.py:309).

### Default event sequence

`_build_events()` gains one entry between `update_exit_branch` and
`ArrivalEvent`:

```
push_temperature → behavior_selection → estuarine_overrides →
update_cwr_counters → MovementEvent → update_exit_branch →
BeenToSeaEvent (NEW) → ArrivalEvent → fish_predation → ...
```

`BeenToSeaEvent` runs after `update_exit_branch` and before
`ArrivalEvent` so the `been_to_sea` write reflects post-movement /
post-exit-branch-teleport cell occupancy, and `ArrivalEvent` reads
the latest `been_to_sea` state.

**Race-condition note** (per pass-1 silent-failure review): C3.3's
`update_exit_branch_id` may teleport stray-strayer agents to
`_branch_entry_cell` of a non-natal branch (delta_routing.py:294).
This mutates `pool.tri_idx` post-movement. `BeenToSeaEvent` reads
the *post-teleport* cell, not the original move-target — which is
correct under round-trip semantics: a strayed agent that's now in a
delta-branch cell did not actually visit a sea cell this step.

### Sim-init validators

Three validators run at `Simulation.__init__`:

1. **Existing C5 `_validate_arrival_event_in_sequence(events)`** —
   already runs after `EventSequencer(self._build_events())`
   (simulation.py:388). Extended to also check `BeenToSeaEvent`:
   on Baltic-detected scenarios, `BeenToSeaEvent` must be present
   AND must come strictly before `ArrivalEvent`. Imports inside
   the method body must be extended to import `BeenToSeaEvent`
   alongside the existing `MovementEvent`/`ArrivalEvent`/`SurvivalEvent`
   imports (simulation.py:471-477).
   - **C5.1's new BeenToSeaEvent missing/misordered checks:
     `RuntimeError` when Baltic-detected.** C4's dormancy guard
     (`raise`) is the precedent. Warning-only would let `arrived ≡ 0`
     (a numerically plausible-looking outcome) hide misconfiguration
     in the legitimate-signal noise.
   - **C5's existing ArrivalEvent missing/misordered checks
     remain warning-only** (preserve current behavior at
     `simulation.py:500-538`). C5 was deliberately soft because
     `pool.arrived` was a soft-tag operation; that posture is
     unchanged. Only the *new* C5.1 BeenToSeaEvent path raises.
   - On non-Baltic scenarios, suppresses C5.1 checks (no warning/raise
     even if `BeenToSeaEvent` absent). C5's existing checks are not
     suppressed (they were Baltic-agnostic before C5.1 and remain so).
2. **New: sea-pen-degenerate-arrival detector.** If
   `_is_baltic` AND any agent's `natal_reach_id ∈ _at_sea_rid_set`,
   emit warning `c5.1-sea-pen-hatchery-degenerate-arrival` at
   logger `salmon_ibm.simulation`. Sea-pen "delayed release" is
   documented Baltic management practice (Buchmann et al. 1997
   — 22,000 Iijoki smolts at Bornholm net-pens; Östergren et al.
   2021 — "delayed release prevents the juveniles from imprinting
   on their natal rivers, drastically increasing straying rates").
   Such configurations make `been_to_sea = True` from t=0 and
   round-trip semantics regress to C5 semantics — degeneracy is
   the correct biological model (no natal river to home to), but
   silent degeneracy is the bug shape we are trying to avoid.
   Warning-only (not raise) because this configuration is
   biologically valid; the user just needs to know `arrived` is
   not measuring round-trip homing for those agents. Cost: ~3
   lines in `_validate_arrival_event_in_sequence`.
3. **Existing C5 missing-event validator** — extended to also
   check `BeenToSeaEvent` per validator 1 above.

**AST sticky-overwrite extension (`tests/test_arrival_event.py`):**

18. **`test_no_event_clears_been_to_sea_to_false`** — AST walker
    extended to assert that any write to `pool.been_to_sea[...]` (or
    `<*>.been_to_sea[...]`) in `events_builtin.py` matches one of
    three explicitly-enumerated sticky-safe AST shapes:

    **(a) Constant `True` literal RHS to subscript assign:**
    ```python
    pool.been_to_sea[mask] = True   # ast.Assign with Constant(True) RHS
    ```

    **(b) Augmented `|=` op to subscript or whole-array:**
    ```python
    pool.been_to_sea |= mask         # ast.AugAssign(op=BitOr)
    pool.been_to_sea[idx] |= mask    # AugAssign on Subscript target
    ```

    **(c) Read-modify-write OR (semantically equivalent to |= but
    parsed as Assign+BinOp):**
    ```python
    pool.been_to_sea[mask] = pool.been_to_sea[mask] | new_mask
    # ast.Assign with BinOp(op=BitOr, left=<same Subscript as target>) RHS
    # The walker must verify the BinOp's left is the same Attribute
    # path as the assign target (not a different attribute).
    ```

    **Reject:** anything else, including:
    - Literal `False` or `0` RHS (the C5 case).
    - Bare boolean-array RHS like `pool.been_to_sea[m] = arr` where
      `arr` is not statically `True` (could contain False elements
      at runtime).
    - `BinOp` RHS with operator other than `BitOr`, or with `left`
      that doesn't match the assign target.
    - Whole-array assignments to a non-`True` literal.

    **Counter-tests** (in same test file, exercising the walker
    directly on synthetic AST nodes — separate from the production-code
    scan):
    - Synthesize `pool.been_to_sea[m] = False` → walker rejects.
    - Synthesize `pool.been_to_sea[m] = True` → walker accepts.
    - Synthesize `pool.been_to_sea |= m` → walker accepts.
    - Synthesize `pool.been_to_sea[m] = pool.been_to_sea[m] | n` →
      walker accepts.
    - Synthesize `pool.been_to_sea[m] = other_pool.been_to_sea[m] | n`
      → walker rejects (left of BinOp doesn't match target).

    **Stronger than C5's existing AST test** (`tests/test_arrival_event.py:446-491`,
    function `test_no_event_clears_arrived_to_false`), which only
    catches literal `False`/`0` constant RHS and would miss
    `pool.been_to_sea[mask] = some_bool_array_with_False_elements`.
    Pass-1 silent-failure review B-4 flagged this gap; pass-2 review
    H-N2 flagged that v2's "True OR |= only" predicate was too
    aggressive (would reject the legitimate read-modify-write OR
    pattern). v3's enumeration covers all three sticky-safe shapes
    and rejects everything else. Implementer should follow C5's
    `ast.parse` + `ast.walk` + `ast.Subscript` inspection pattern
    but extend the predicate per the enumeration above.

## §5 Edge cases

**Hatchery release in BalticCoast / OpenBaltic ("delayed release").**
Documented Baltic management practice — Buchmann et al. 1997 (10.1016/S0044-8486(97)00097-5,
verified scite 2026-05-08) describes 22,000 Iijoki Baltic salmon
smolts transferred to net-pens off Bornholm at 8 ppt; Östergren et
al. 2021 (10.1098/rspb.2020.3147, verified scite 2026-05-08): "so-called
delayed release (i.e. fish stocked in the sea rather than in rivers
to increase their survival rate) aimed at supporting salmon sea
fisheries. Delayed release prevents the juveniles from imprinting on
their natal rivers, drastically increasing straying rates." For such
agents, `been_to_sea = True` at t=0 and round-trip semantics regress
to C5 semantics — which is *biologically correct* for these agents
(they have no natal-river imprinting to home to). The metric just
isn't measuring "round-trip homing" for them. Sim-init validator (§2
validator 2) emits a warning so the user knows. **In-scope; not OOS.**

**Agent dies at sea.** Sticky flag stays True post-mortem. `ArrivalEvent`'s
`pool.alive` check filters them out of arrival anyway. The asymmetry
is "dead-and-True stays True; dead-and-False stays False" — correct
under sticky semantics. No special handling needed.

**Agent oscillates between sea and lagoon.** Each at-sea visit
re-sets `been_to_sea = True` (idempotent under sticky). No issue.

**Non-Baltic landscapes (Columbia, TriMesh).** `_is_baltic = False`,
`_at_sea_rid_set = frozenset()`, `BeenToSeaEvent.execute` no-ops,
sim-init validators suppressed. **Acceptance criterion**: pre-C5.1
non-Baltic runs that produced `arrived > 0` under C5 must continue
to do so. Verification path (Plan task — not yet executed):

```python
# Confirm no non-Baltic config produces non-empty thresholds:
for cfg in ['config_columbia', 'config_curonian_trimesh', ...]:
    sim = Simulation(load_config(cfg), ...)
    assert sim._arrival_threshold_by_natal_rid == {}, \
        f"non-Baltic {cfg} has non-empty thresholds — C5.1 guard would break it"
```

If verification fails for any non-Baltic config, the `ArrivalEvent`
guard must be conditionally gated (`if pool_at_least_partially_in_baltic:
apply guard else: skip guard`) — defensive code path. If verification
passes (expected — the C5 thresholds depend on `mesh.dist_from_sea`
which only the Baltic H3 multires NC carries), no defensive code needed.

**Post-teleport `pool.tri_idx` race.** `update_exit_branch_id`
(delta_routing.py:256-309) may teleport stray-strayer agents into
non-natal delta-branch cells (line 294 path B). `BeenToSeaEvent`
runs after, reading the *post-teleport* cell. Behavior:
- Strayer teleported INTO at-sea cell (rare but possible if a
  delta branch's `_branch_entry_cell` happened to have reach_id =
  BalticCoast — should not occur in a correct mesh): `been_to_sea`
  set this step. Correct.
- Strayer teleported OUT OF at-sea cell into delta-branch cell:
  `been_to_sea` NOT set this step (lost the at-sea visit signal
  for this step). However, since `been_to_sea` is sticky, if the
  agent had been at sea at any prior step, it stays True. This
  means a strayed-from-sea agent loses the *current step's* sea
  signal but retains *prior steps*' signal. Acceptable — the
  semantic is "ever been at sea", not "currently at sea".

**Mesh `reach_id` boundary cells.** Cell membership is a discrete
categorisation per-cell-id; no boundary issue *unless* cells are
miscategorised. Verified that the build script
`scripts/build_h3_multires_landscape.py` assigns `reach_id` from
polygon containment, with delta-branch cells categorised by
branch-name not by sea/coast — no overlap. `BalticCoast` and
`OpenBaltic` cells are non-overlapping by construction.

## §6 Out of scope

- **B3 (salinity-threshold sea-marker).** Considered and rejected at
  brainstorm time: adds a tunable knob without prior literature
  anchor, while published Baltic salmon studies operationalize "at
  sea" geographically. If future work needs physiologically-anchored
  smoltification triggers, that's a separate spec.
- **Multi-year simulation horizon.** The 480h default scenario is
  smolt-outmigration; full lifecycle homing return needs months-to-years
  of sim time. C5.1 does not change the time-horizon defaults; that's
  a separate calibration concern. C5.1 ships test #20 as a
  long-horizon smoke (8760h) but keeps it `@pytest.mark.slow`.
  **Plan must include explicit step**: run the slow suite manually
  before tagging release (`pytest tests/ -m slow -v`). Otherwise
  test #20 silently rots when the default CI lane skips slow tests
  and the §8.12 acceptance criterion is never actually validated.
- **Per-origin homing-return contrast (C5.2 / C6).** C5.1 makes
  `pool.arrived` meaningful for any agent. Splitting `arrived` rate
  by `pool.origin` (wild vs hatchery) is the publication-grade
  signal that C1+C2+C3.1+C3.2+C3.3+C4+C5+C5.1 collectively produce.
  Per-origin reporting (CSV column, UI badge, scite-anchored
  validation against literature numerical ranges) is C5.2 / future
  tier. Calibration anchors for that work:
  - **Larsson 1984** (10.3354/meps017215, *Marine Ecology Progress
    Series*, verified scite 2026-05-08) — 210,000 recaptures from
    1.7M tagged Swedish hatchery smolts ≈ **~12% gross recapture
    rate** (sea + river combined). Canonical Baltic figure.
  - **Jokikokko et al. 2006** (10.1111/j.0022-1112.2006.00892.x,
    verified scite 2026-05-08) — Simojoki recapture rates: **wild
    1.7–17.0%, semi-wild 1.0–13.1%, hatchery-as-smolt 1.3–6.3%**
    (totals across sea, coastal, and river gear).
  - **Insulander & Ragnarsson 2001** (10.1046/j.1365-2400.2001.00177.x,
    verified scite 2026-05-08) — Dalälven hatcheries 9–10 km from
    the river mouth; differential return-to-river fractions
    (qualitative comparative ratio narrative; per-origin return-rate
    percentages would require retrieving the full text — abstract
    alone is comparative-directional, not absolute).
- **Iteroparous repeat-spawner re-arrival semantics.** Atlantic
  salmon are partially iteroparous (Persson et al. 2023,
  doi:10.1139/cjfas-2022-0126, verified scite 2026-05-08; documents
  0–26% repeat-spawners across rivers, average 3.8%). C5.1's
  sticky-once-set `arrived` cannot distinguish first-arrival from
  repeat-arrival. Forward-pointer for future work that needs to
  count return events, not just return state.
- **Per-origin straying-rate framing.** `straying_rate(origin) = 1
  - arrived(origin)/n_alive(origin)`. Östergren et al. 2021 frames
  straying as the headline metric *qualitatively* ("drastically
  increasing straying rates" via delayed release; the quote is
  discussion-section, not a numerical anchor — Carlin 1969 is the
  underlying primary source for actual rates). Vasemägi et al. 2005
  frames it as immigration into the natal river (inverse;
  quantitative: 0.068 baseline / 0.249 peak). The duality is worth
  surfacing in the C5.2 reporting spec. Not part of C5.1.
- **Sea-pen hatchery configuration validator escalation.** v2 ships
  the warning-only detector (§2 validator 2). Future work could
  raise on misconfigured sea-pen+wild-natal combinations, but the
  current biological semantics treat any `natal_reach_id ∈
  _at_sea_rid_set` agent as "definitionally degenerate" with no
  bug to surface — warning is sufficient.

## §7 Empirical anchors

**Strictness defence.** The literature on Baltic salmon "at-sea"
operationalization is heterogeneous: Vasemägi et al. 2005 sampled
spawners *in-river*, Insulander & Ragnarsson 2001 measured
*return-to-river* via fish-trap, Jokikokko et al. 2006 reports
recapture rates *integrated across sea, coastal, and river gear*.
None of these papers operationalize "at sea" as "in offshore Baltic
basin proper". C5.1 chooses the **strictest geographic operationalization
the literature can support** (`been_to_sea = True` requires occupying
a BalticCoast or OpenBaltic cell, not just delta-branch entry)
because **false-positives — counting a fish that never genuinely
left the river system as "homed back" — are more publication-damaging
than false-negatives** — missing a fish that briefly entered coastal
waters but didn't get tagged because the simulation horizon was too
short. The choice errs on the strict side; reviewers may argue with
the choice but cannot argue we hid the heterogeneity.

**B2 vs B1 (rejected) defence.** B1 was "entered delta branch =
been to sea". Rejected as a **category error** for the Curonian
system: delta branches are *river mouths feeding the lagoon*, not
sea. An agent that swims into a delta branch and turns back has
"started seaward journey" but is still in oligohaline lagoon water,
biologically not at sea. The Källo, Baktoft & Kristensen 2022 study
that operationalized return at the river-mouth scale was on
*anadromous brown trout (Salmo trutta) in a Danish fjord* — different
species, different topology — and is not a precedent for Baltic
salmon at the Curonian-to-Baltic scale.

**Verified citations** (all DOI-resolved via scite MCP, 2026-05-08):

- **Vasemägi, Gross & Paaver 2005** (doi:10.1038/sj.hdy.6800693) —
  Vindelälven 18-yr genetic mixed-stock analysis. Methodology:
  fin-damaged spawners (n=181) caught **in River Ume/Vindelälven**,
  1997–2003. Provides hatchery-vs-wild straying rate evidence (0.068
  baseline / 0.249 peak immigration). Already C3.3 anchor.
  (Note: spec v1 incorrectly framed this as "Baltic-coast caught";
  corrected v2.)
- **Insulander & Ragnarsson 2001** (doi:10.1046/j.1365-2400.2001.00177.x) —
  Dalälven release-site fidelity: hatcheries 9–10 km from river
  mouth; differential return-to-river fractions. Already C3.3
  anchor; supports release-site = `natal_reach_id` model.
- **Jokikokko, Kallio-Nyberg & Saloniemi 2006**
  (doi:10.1111/j.0022-1112.2006.00892.x) — Simojoki recapture rates
  reported as totals across sea/coastal/river gear. Already C3.3
  anchor; quantitative reference for §6 deferred work.
- **Buchmann, Dalsgaard & Nielsen 1997**
  (doi:10.1016/S0044-8486(97)00097-5) — describes 22,000 Iijoki
  Baltic salmon smolts at Bornholm net-pens; the empirical anchor
  for §5 sea-pen edge-case being real Baltic management practice.
  New for C5.1.
- **Östergren, Palm & Gilbey 2021**
  (doi:10.1098/rspb.2020.3147) — "delayed release prevents the
  juveniles from imprinting on their natal rivers, drastically
  increasing straying rates." Reinforces sea-pen edge-case scope and
  provides §6 straying-rate framing. New for C5.1.
- **Friedland, Dannewitz & Romakkaniemi 2016**
  (doi:10.1093/icesjms/fsw178) — post-smolt distribution analysis;
  predation interactions concentrated in coastal waters. Empirical
  anchor for §2 BalticCoast inclusion. New for C5.1.
- **Jacobson, Gårdmark & Huss 2019**
  (doi:10.1111/jfb.14213) — five-decade post-smolt distribution
  analysis. Reinforces BalticCoast inclusion. New for C5.1.
- **Larsson 1984** (doi:10.3354/meps017215) — canonical Baltic
  salmon recapture rate (~12% gross). Calibration target for §6
  deferred per-origin work. New for C5.1.

**Removed from v1 spec:** Källo, Baktoft & Kristensen 2022
(10.1093/icesjms/fsac079) — pass-1 review found the paper is on
*Salmo trutta* (anadromous brown trout) in a Danish fjord with
PIT-tag river-mouth detection, not Atlantic salmon. The v1 spec
mischaracterised the paper. Removed as part of
v2.

## §8 Acceptance criteria

1. `AgentPool.been_to_sea` exists, defaults False, round-trips through
   `compact()` and `add_agents()`. `ARRAY_FIELDS` list is canonical.
   `agents.py:88-96` init guard asserts the field is present.
2. `BeenToSeaEvent` is registered via `@register_event("been_to_sea")`
   and is vectorised (no Python-level agent loop). Uses bare
   `population.pool` (no `getattr` shim) and `np.int32`-cast
   `cell_rid` (defends against int8 overflow at reach index ≥128).
3. `BeenToSeaEvent.execute` is a no-op when `_is_baltic = False`;
   never raises on non-Baltic landscapes.
4. `pool.been_to_sea` is sticky: AST test (§4.18) confirms no event
   in `events_builtin.py` writes anything other than literal `True`
   or `|=` augmented assignment to it.
5. `ArrivalEvent`'s arrival mask includes the `pool.been_to_sea`
   guard. Pre-existing agents-in-natal-upper-quartile satisfy the
   arrival predicate iff they have visited a BalticCoast or
   OpenBaltic cell.
6. `_is_baltic` is a positive Baltic-detection signal independent of
   any single reach name lookup. Conjunction:
   `(isinstance(_species_config.wild, BalticBioParams) OR
   isinstance(_species_config.hatchery, BalticBioParams))`
   AND `any BRANCH_FRACTIONS key in mesh.reach_names`. The
   wild-OR-hatchery disjunction handles configs where the Baltic
   species_config arrives via the hatchery side only. No single
   reach-name typo can silently flip Baltic→non-Baltic. Note
   private-attribute access (`_species_config`, NOT
   `species_config`); this matches the existing convention at
   `simulation.py:302`.
7. `_at_sea_rid_set` is a `frozenset[int]` containing the reach IDs
   for "BalticCoast" and "OpenBaltic" if `_is_baltic`, else empty.
8. **`RuntimeError` raised at sim init** if `_is_baltic` and (a)
   `BeenToSeaEvent` missing, (b) `BeenToSeaEvent` after
   `ArrivalEvent`, or (c) neither "BalticCoast" nor "OpenBaltic" in
   `mesh.reach_names`. Distinct err-id per case. **Existing C5
   ArrivalEvent missing/misordered checks remain warning-only**
   — only the new C5.1 BeenToSeaEvent path raises.
9. **Warning at sim init** if `_is_baltic` and any agent's
   `natal_reach_id ∈ _at_sea_rid_set` (sea-pen-degenerate); does
   NOT raise. Additionally, **partial-miss warning** if `_is_baltic`
   and only ONE of {"BalticCoast", "OpenBaltic"} resolves in
   `mesh.reach_names` (warns per missing name; does not raise — the
   partial set is still a usable round-trip filter).
10. Default `_build_events()` sequence puts `BeenToSeaEvent` between
    `update_exit_branch` and `ArrivalEvent`.
11. Default-scenario smoke (rng_seed=42, n_agents=50, n_steps=480h):
    `arrived ∈ [0, n_alive]`, **strictly less than the C5-baseline
    arrived count** at identical parameters (i.e., strictly less
    than the v1.7.10 reading of 46). Specific value is parameter-
    dependent and not part of acceptance — `arrived = 0` at this
    horizon is a *valid* result, not a test failure (see §1).
    Expressed relatively rather than as the absolute "< 46" to
    avoid brittleness if future swim-speed parameter changes
    legitimately shift the C5 baseline.
12. Long-horizon smoke (rng_seed=42, n_agents=200, n_steps=8760h,
    `@pytest.mark.slow`): `arrived > 0` for at least one origin
    class. Validates the round-trip mechanic fires when given
    biologically-realistic time.
13. Full pytest sweep: baseline (C5.1 lands on top of v1.7.10 = 925
    passed) + 18 new tests (12 unit in `test_been_to_sea_event.py`
    + 5 validator + 1 AST sticky in `test_arrival_event.py`) + 1
    slow-test (#20, skipped by default) = **~944 collected**, all
    passing, 1 deselected (slow). The slow test counts toward
    collected but not toward passed-on-default-CI; manual `pytest -m
    slow` run before tagging release exercises §8.12 acceptance.
14. UI dashboard at `https://laguna.ku.lt/HexSimPy/` after deploy:
    `arrived` reads strictly less than `alive` for the default
    scenario (typically 0 at 480h horizon), confirming the
    round-trip filter is active.
