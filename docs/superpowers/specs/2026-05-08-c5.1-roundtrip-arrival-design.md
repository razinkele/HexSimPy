# C5.1 — Round-trip Arrival Semantics

**Date:** 2026-05-08
**Owner:** @razinkele
**Status:** 🟡 DRAFT v1 — awaiting user review and review-loop convergence.

C5.1 tightens C5's arrival semantics from "in upper-quartile of natal
reach" (which counts both never-left agents and returned agents) to
"returned to upper-quartile of natal reach **after a verified sea
visit**" (which counts only the round-trip homing event). Adds one
boolean column to `AgentPool` (`been_to_sea`) and one new event
(`BeenToSeaEvent`); guards `ArrivalEvent`'s arrival mask with the new
column.

---

## §1 Why

C5 (v1.7.9) wired up `pool.arrived` after years of dead state. v1.7.10's
Playwright verify on laguna confirms the engine works — but exposes that
the metric is degenerate at this simulation horizon:

> Default scenario, rng_seed=42, n_agents=50, n_steps=480h:
> **46/50 alive · 46 arrived**, exit_branch_tagged=4.

Every alive agent reads as "arrived". Reason: spawning grounds are
intrinsically upstream-located, so agents start in cells that already
sit in the upper quartile of their natal reach. The 75th-percentile
threshold is a near-tautology with "alive in natal reach". The signal
the publication-grade four-tier hatchery-vs-wild architecture is
designed to produce is **homing return rate** (fraction of released
smolts that complete the migration loop and home back). C5 cannot
distinguish "started here, never left" from "left, came back".

C5.1 makes the metric biologically meaningful by requiring evidence of
a sea visit before an agent qualifies as arrived. After C5.1 the
counter at this horizon will read low (likely 0–4 in a 480h smolt-out
simulation, given that few agents reach OpenBaltic in 20 days) — that's
the *correct* signal for the simulation horizon, not a bug.

## §2 Design

### Architecture

Round-trip semantics implemented as a **two-state guarded sticky flag**:

```
been_to_sea[i] := True iff agent i has, at some t' ≤ t, occupied a cell
                       whose reach_id corresponds to "OpenBaltic".
arrived[i]     := True iff (been_to_sea[i]
                            AND in upper-quartile-of-natal-reach[i]).
```

Both flags are sticky-once-set; neither is ever cleared by any event.

### New state: `AgentPool.been_to_sea`

- `np.ndarray[bool]` of length `n_agents`, defaults all-`False` at init.
- Added to `AgentPool.ARRAY_FIELDS` so it round-trips through
  `compact()`, `add_agents()`, and the OOP `AgentView`.
- Mirrors `pool.arrived`'s shape and semantics exactly.

### New event: `BeenToSeaEvent`

Registered via `@register_event("been_to_sea")` in `events_builtin.py`.
Vectorised; reads agent's current cell `reach_id`, compares to the
sim's pre-resolved `_open_baltic_rid`, sets `been_to_sea[i] = True`
where the cell-reach-id matches and `alive[i]` is True. Never clears.

```python
# Sketch (full code in plan):
class BeenToSeaEvent(Event):
    name: str = "been_to_sea"

    def execute(self, population, landscape, t, mask):
        sim = landscape["sim"]
        rid = sim._open_baltic_rid
        if rid < 0:
            return  # non-Baltic landscape; no-op
        pool = getattr(population, "pool", population)
        cell_rid = sim.mesh.reach_id[pool.tri_idx]
        at_sea_now = (cell_rid == rid) & pool.alive
        # Sticky: only set True, never clear
        pool.been_to_sea[at_sea_now] = True
```

### Guarded `ArrivalEvent`

`ArrivalEvent.execute` gains one line:

```python
arrived_now = (
    in_upper_quartile_mask
    & pool.alive
    & pool.been_to_sea           # ← C5.1: round-trip guard
    & ~pool.arrived              # don't re-tag (sticky)
)
```

### Sim-init resolution: `_open_baltic_rid`

`Simulation.__init__` resolves the OpenBaltic reach ID once at startup:

```python
try:
    self._open_baltic_rid = int(self.mesh.reach_names.index("OpenBaltic"))
except (AttributeError, ValueError):
    self._open_baltic_rid = -1  # non-Baltic landscape
```

Wired in `simulation.py` after `assert_branch_topology(self.mesh)` and
before `_compute_arrival_thresholds()`. The `-1` sentinel is the same
"no-op for non-Baltic" pattern C3.3 uses.

### Default event sequence

`_build_events()` gains one entry between `update_exit_branch` and
`ArrivalEvent`:

```
push_temperature → behavior_selection → estuarine_overrides →
update_cwr_counters → MovementEvent → update_exit_branch →
BeenToSeaEvent (NEW) → ArrivalEvent → fish_predation → ...
```

`BeenToSeaEvent` must run after movement (so the agent's current cell
reflects the latest step) and before `ArrivalEvent` (so the guard
reads the latest sea state).

### Sim-init validator

`Simulation._validate_arrival_event_in_sequence` (added in C5) gains a
new check: if the scenario is Baltic (`_open_baltic_rid >= 0`) and
`ArrivalEvent` is in the sequence, `BeenToSeaEvent` must be present
and must come strictly before `ArrivalEvent`. Emits warning at logger
`salmon_ibm.simulation` with new err-id `c5.1-been-to-sea-missing` or
`c5.1-been-to-sea-misordered`.

## §5 Edge cases

**Hatchery release in OpenBaltic.** If a future config releases hatchery
agents directly into an OpenBaltic cell (sea-pen aquaculture), they
would have `been_to_sea = True` from t=0 and round-trip semantics
degenerate to C5 semantics for those agents. This is a configuration
concern, not a model bug — the `arrived` count for such agents
correctly reads "in natal upper quartile while been_to_sea is True"
which is degenerate iff the natal reach is OpenBaltic. Document but
don't guard.

**Agent dies at sea.** Sticky flag stays True. `ArrivalEvent`'s
`pool.alive` check filters them out of arrival anyway. Correct.

**Agent oscillates between sea and lagoon.** Each at-sea visit re-sets
`been_to_sea = True` (idempotent under sticky). No issue.

**Non-Baltic landscapes (Columbia, TriMesh).** `_open_baltic_rid = -1`,
`BeenToSeaEvent.execute` no-ops. `ArrivalEvent`'s guard makes
`arrived` permanently stuck at 0 — same effective behavior as
pre-C5 (acceptable for non-Baltic; the round-trip semantic is
Curonian-specific). Sim-init validator suppresses the missing-event
warning for non-Baltic.

**`pool.arrived` already True from C5 semantics in pre-existing
saved state.** Not applicable — `pool.arrived` is reset at
`Simulation.__init__`. No saved-state recovery scenarios in this IBM.

## §6 Out of scope

- **B3 (salinity-threshold sea-marker).** Considered and rejected at
  brainstorm time: adds a tunable knob without prior literature
  anchor, while published Baltic salmon studies operationalize "at
  sea" geographically. If future work needs physiologically-anchored
  smoltification triggers, that's a separate spec.
- **Multi-year simulation horizon.** The 480h default scenario is
  smolt-outmigration; full lifecycle homing return needs months-to-years
  of sim time. C5.1 does not change the time-horizon defaults; that's
  a separate calibration concern.
- **Per-origin homing-return contrast.** C5.1 makes `pool.arrived`
  meaningful for *any* agent. Splitting `arrived` rate by
  `pool.origin` (wild vs hatchery) is the publication-grade signal
  that C1+C2+C3.1+C3.2+C3.3+C4+C5+C5.1 collectively produce, but the
  per-origin reporting itself (CSV column, UI badge, scite-anchored
  validation against Insulander 2001 / Vasemägi 2005 numerical
  ranges) is C5.2 / C6 / future tier.
- **Hatchery-release-in-OpenBaltic configuration validator.** Could
  add a sim-init check rejecting configurations where the hatchery
  release-site reach == OpenBaltic. Out of scope unless someone
  proposes such a config.

## §7 Empirical anchors

The B2 operationalization ("at sea" = "in OpenBaltic cell") is
congruent with the Baltic salmon literature already cited by prior
tiers:

- **Vasemägi, Gross & Paaver 2005** (doi:10.1038/sj.hdy.6800693) —
  Vindelälven 18-yr genetic mixed-stock analysis based on Baltic-coast
  caught salmon. C3.3 anchor; "at sea" defined geographically as
  Baltic basin. Direct precedent for the B2 operationalization.
- **Insulander & Ragnarsson 2001** (doi:10.1046/j.1365-2400.2001.00177.x) —
  Dalälven release-site fidelity: counts at *return-to-river* after a
  sea phase, not at delta-branch entry. Direct precedent for
  round-trip semantics.
- **Källo, Baktoft & Kristensen 2022** (doi:10.1093/icesjms/fsac079) —
  imprinting-timing study; uses sea-recapture as the post-smolt
  validation milestone. Supports B2 over B1.
- **Jokikokko, Kallio-Nyberg & Saloniemi 2006**
  (doi:10.1111/j.0022-1112.2006.00892.x) — Simojoki recapture rates
  reported as fraction *recovered at sea*, not fraction *that left
  the river*. Reinforces B2.

No new citations needed for C5.1 itself — it's a tightening of the
operationalization, not a new biological claim. All four anchors
above are already in `docs/scientific-foundations-hatchery-wild.md`
and DOI-verified per the project's scite-verification protocol.

## §8 Acceptance criteria

1. `AgentPool.been_to_sea` exists, defaults False, round-trips through
   `compact()` and `add_agents()`. ARRAY_FIELDS list is canonical.
2. `BeenToSeaEvent` is registered via `@register_event("been_to_sea")`
   and is vectorised (no Python-level agent loop).
3. `BeenToSeaEvent.execute` is a no-op on non-Baltic landscapes
   (`_open_baltic_rid == -1`); never raises.
4. `pool.been_to_sea` is sticky: AST test confirms no event in
   `events_builtin.py` writes `False` or `0` to it.
5. `ArrivalEvent`'s arrival mask includes the `pool.been_to_sea`
   guard. Pre-existing agents-in-natal-upper-quartile still satisfy
   the arrival predicate iff they have visited OpenBaltic.
6. Sim-init validator emits warnings for `c5.1-been-to-sea-missing`
   and `c5.1-been-to-sea-misordered` on Baltic scenarios; suppresses
   warnings on non-Baltic scenarios.
7. Default `_build_events()` sequence puts `BeenToSeaEvent` between
   `update_exit_branch` and `ArrivalEvent`.
8. End-to-end default-scenario smoke (rng_seed=42, n_agents=50,
   n_steps=480h): `arrived ≤ exit_branch_tagged`. (Strict inequality
   would also be acceptable; equality means every delta-toucher
   reached OpenBaltic and returned.)
9. Full pytest sweep: baseline (C5.1 lands on top of v1.7.10 = 925
   passed) + ~13 new tests = ~938 collected, all passing.
10. UI dashboard at `https://laguna.ku.lt/HexSimPy/` after deploy:
    `arrived` reads strictly less than `alive` for the default
    scenario, confirming the round-trip filter is active.
